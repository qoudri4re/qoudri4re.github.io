Database per tenant for a multi tenancy system

May 21, 2026 ·

I am currently working on a multi-tenant project where I had to choose the right tenancy database pattern. Row-level tenant separation was one of the obvious options, but it was not a good fit because the system could end up handling a very large amount of records, and that growth would not be evenly distributed across tenants.

That approach is popular because it is easy to start with:

  • one database
  • one schema
  • one set of tables
  • every row carries a tenant_id

It is convenient, but the tradeoffs won't work for the project's requirements.

Limits Of Row-Level Separation

The biggest problem is the noisy neighbour effect.

If every tenant shares the same database tables, one large tenant can affect everyone else:

  • heavy queries from one tenant can slow down others
  • index growth and table bloat affect the whole system
  • maintenance tasks hit all tenants together
  • one bad query pattern becomes everybody's problem

There is also the constant risk of tenant-filter mistakes.

With row-level separation, every query depends on getting this right: 

WHERE tenant_id = ?

Miss that filter once, or apply it incorrectly in one edge case, and you have a data leak.

Limits Of Schema-Level Separation

The next obvious option is schema-per-tenant.

That solves some of the row-level leakage problems because each tenant gets its own schema inside the same database.

However, Schema-level multi-tenancy comes with its own downsides:

  • all tenants still compete for the same database server resources
  • a noisy neighbour can still hurt performance
  • migrations become more annoying because you need to apply them across many schemas
  • connection and search path management gets more complex
  • tenants are still not as isolated as they would be with fully separate databases

It is definitely better isolation than row-level separation, but it still keeps everyone living in the same house.

Database-Per-Tenant

So I chose a different model:

  • one registry database for shared platform-level records
  • one PostgreSQL database per tenant for tenant-owned data

That gives much better isolation:

  • one tenant's data is physically separated from another's
  • backup and restore can happen per tenant
  • failures have a smaller blast radius
  • the risk of accidental cross-tenant queries drops dramatically

But this approach is not free either.

The main disadvantages are:

  • more connection management
  • connection pools can grow aggressively as tenant count grows
  • more operational complexity
  • tenant-aware routing becomes an application concern
  • migrations and setup have to run across many databases
  • the ORM can no longer assume there is one global database connection

That connection-pool issue is a big one.

With database-per-tenant, every active tenant may need its own Sequelize instance, and every Sequelize instance has its own connection pool. That means the scaling problem is not only "more tenants means more data". It is also "more tenants means more pools".

At 20 tenants, that is manageable. At 500 tenants, it becomes something you have to design around very deliberately.

Even if each tenant only keeps a small pool, the total number of open or recently-opened connections can become expensive.

As a rough assumption, if each tenant-scoped connection setup ends up consuming even a modest amount of memory and connection overhead, the cost adds up faster. At that point, tenant growth is not just about more rows in, it is also about more connection objects, more pool bookkeeping, and more pressure on both the Node process and Postgres.

In practice, that means:

  • more memory usage in the app process
  • more open database connections to manage
  • more idle connections hanging around unless you clean them up
  • more pressure on the database server if many tenants become active around the same time

That is one of the biggest drawbacks of this approach, and it is exactly why a connection manager with caching and idle cleanup is not optional here.

It is also why I plan to put PgBouncer in front of Postgres.

PgBouncer helps absorb some of the connection turnover and reduces how expensive it is for many tenant-specific pools to talk to Postgres directly. It helps manage the problem, but it does not eliminate the architectural reason the problem exists. The app can still end up with many tenant-scoped Sequelize instances and pools living in memory.

Realistically, I do not expect this project to hit some wild tenant count. It may never even reach 100 active tenants. And even if it eventually grows far beyond that, this is the kind of problem that can usually be solved with infrastructure and scaling.

More specifically, the need for tenant-aware connection handling and request-time database routing is what shaped the rest of the implementation.

Once each tenant has its own database, the question becomes:

How do you make sure every incoming request uses the correct tenant database, every time?

The Problem With Sequelize in this Multi-Tenant Setup

The issue comes from how model classes hold on to their Sequelize instance.

Normally, a model like TenantAdmin has a static .sequelize property pointing to the connection it was initialized with. That works fine in a single-tenant app.

It breaks the moment multiple tenant connections exist.

Imagine this:

Tenant A connection starts
TenantAdmin.sequelize -> tenantA

Tenant B connection starts later
TenantAdmin.sequelize -> tenantB

Now every part of the app using TenantAdmin is pointing at Tenant B.

This would cause wrong data, or empty results, or subtle cross-tenant corruption.

The Fix: AsyncLocalStorage

The approach I used in this project is built around AsyncLocalStorage.

AsyncLocalStorage is a Node.js feature for storing request-specific data across an async call chain.

The easiest way to think about it is this: once a value is attached to a request's execution flow, any function running inside that same flow can read it later, even after several awaits.

That is exactly what this multi-tenant problem needs.

The system has to remember:

  • which tenant this request belongs to
  • which tenant database connection should be used

and it has to remember that not just in one controller function, but across all the nested async work that happens underneath it.

The idea is:

  • incoming requests that require database access get bound to the correct tenant database
  • every async call inside that request inherits that database context
  • every model reads from the current async context instead of a stale static pointer

So instead of:

TenantAdmin.sequelize;

as a long-lived global value, it resolves dynamically from the active request context.

The Three Pieces That Make It Work

1. Patch the model classes once at startup

The patch is a runtime change to an existing property on each model class.

In JavaScript, Object.defineProperty(...) lets you redefine how a property behaves. The code takes the static .sequelize property on every tenant-scoped model and replaces it.

Before the patch, .sequelize behaves like a normal stored value: 

TenantAdmin.sequelize = someDatabaseConnection;

After the patch, .sequelize is no longer treated as a simple value sitting on the model class. It becomes a computed property.

This is achieved by bringing in the getter.

A getter is a function that runs automatically when a property is read. So when code does this: 

TenantAdmin.sequelize;

JavaScript does not just return a previously stored value. It calls the getter function and uses whatever that function returns.

In this case, the getter asks AsyncLocalStorage for the database bound to the current request: 

const db = storage.getStore();

So the flow is:

  1. some ORM operation tries to read Model.sequelize
  2. the getter runs
  3. the getter asks AsyncLocalStorage for the active tenant DB
  4. that tenant DB is returned

If there is no active DB context, it throws immediately.

The patched version looks like this:

Object.defineProperty(ModelClass, "sequelize", {
  get() {
    const db = storage.getStore();
    if (!db) {
      throw new Error(
        `${ModelClass.name}.sequelize accessed outside a runWithDb context`,
      );
    }
    return db;
  },
  set(_value) {
    // no-op
  },
});

The small but important detail is the set(_value) part.

Normally, Sequelize may try to assign to .sequelize again when models are initialized on another connection. In a database-per-tenant system, that would reintroduce the same problem I was trying to avoid.

So the setter is turned into a no-op. In plain terms, writes to .sequelize stop mattering, while reads to .sequelize always go through the getter.

So the patch:

  • stop trusting a shared stored value
  • replace it with a computed lookup
  • make every model resolve its database from the active request context

2. Use a connection manager

Once each tenant has its own database, opening a brand new Sequelize connection for every request would be wasteful, hence the need for a connection manager.

Its job is to:

  • cache tenant-specific Sequelize connections
  • reuse them when the same tenant sends another request
  • lazily create a connection only when needed
  • close idle connections after some time

Without a connection manager, database-per-tenant would quickly become more expensive than it needs to be.

The pattern looks roughly like this:

const cached = this.connections.get(tenantId);

if (cached) {
  cached.lastActivity = Date.now();
  return cached.sequelize;
}

const tenant = await Tenant.findOne({ where: { tenantId, isActive: true } });
const sequelize = new Sequelize({ database: tenant.dbName, ... });

await sequelize.authenticate();
await runWithDb(sequelize, async () => {
  sequelize.addModels(ALL_TENANT_MODELS);
});

this.connections.set(tenantId, {
  sequelize,
  lastActivity: Date.now(),
});

So the connection manager is really the operational half of this architecture. AsyncLocalStorage solves routing inside a request. The connection manager solves how those tenant connections are created, reused, and cleaned up.

3. Bind the right database per request

When a request comes in, I determine which tenant it belongs to, by asking the connection manager for the matching connection, and then wrapping the rest of the request in: 

runWithDb(db, () => next());

That means everything downstream runs with that tenant DB in scope.

4. Provide an escape hatch

Sometimes I may need the active DB instance directly, especially for transactions. For that, the project exposes: 

getDb();

But only inside a valid tenant context. Outside that, it throws.

How Request Routing Works

There are two main request paths in the system.

Unauthenticated routes

For no auth routes, there is no JWT yet. So the tenant is resolved from the subdomain.

In development, that comes from the tenant subdomain header. In production, it comes from the host.

Once the tenant is identified, the app loads that tenant connection and binds it to the request context before the controller runs.

That flow looks like this:

router.post(
  "/login",
  asyncHandler(resolveTenant),
  asyncHandler(tenantAdminLogin),
);

And inside the middleware:

const db = await connectionManager.getConnection(tenantId);
return runWithDb(db, () => next());

Authenticated routes

For authenticated routes, the JWT already contains the tenantId.

So the flow is:

  1. auth middleware validates the token
  2. it stores tenantId on the request
  3. attachTenantDb fetches the right tenant connection
  4. runWithDb(db, () => next()) binds that DB for the rest of the request

By the time the controller executes, tenant models are automatically pointing at the right tenant database.

That flow looks like this:

router.get("/", adminAuthMiddleware, attachTenantDb, asyncHandler(listRecords));

And attachTenantDb does the actual tenant binding: 

export async function attachTenantDb(
  req: AuthRequest,
  res: Response,
  next: NextFunction,
) {
  const tenantId = req.tenantId;
  const db = await connectionManager.getConnection(tenantId);
  req.tenantDb = db;
  return runWithDb(db, () => next());
}

After that, normal model queries work the way you would expect: 

export async function listRecords(req: AuthRequest, res: Response) {
  const records = await Record.findAll();
  return ResponseHandler.success(res, records);
}

No tenant DB is passed into the controller manually. The active database comes from the async context.a

There is one more practical implication here: any code that needs a transaction inside a tenant-scoped request must obtain it from the active context, hence one of the many reasons why getDb() exists.
